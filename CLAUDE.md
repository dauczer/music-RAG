# 🎤 Artist DNA — CLAUDE.md

## Project overview
A RAG-powered chatbot that knows an artist's entire universe through their lyrics (via Genius API).
Users can ask questions like:
- "How did Damso's themes evolve between Ipséité and Qalf?"
- "What are Nekfeu's most recurring lyrical obsessions?"

This is a **portfolio project** demonstrating: data ingestion, 
chunking strategy, vector embeddings, ChromaDB, and a FastAPI backend.

---

## Tech stack
| Role | Tool | Notes |
|---|---|---|
| Lyrics | `requests + BeautifulSoup` | Genius API + HTML scraping |
| Embeddings | `sentence-transformers` | Runs locally, free, no API needed |
| Vector DB | `ChromaDB` | Local persistent storage |
| LLM | `google-generativeai` (Gemini Flash) | Free tier, 1500 req/day |
| Orchestration | `LangChain` | RAG chain only |
| Backend | `FastAPI` | One `/chat` endpoint |
| UI | Your Svelte portfolio | Chat window calls the FastAPI endpoint |

## Implementation order
Build and **test each step before moving to the next.** Do not build everything at once.

### Phase 1 — Data ingestion
1. Build `ingestion/genius_scraper.py`
   - Function `scrape_artist(artist_name, max_songs=200)`
   - Fetches all primary songs from Genius API, shuffles, scrapes lyrics via BeautifulSoup
   - Skips unofficial snippets (title ends with `*`) and songs with < 600 chars
   - Saves to `data/raw/{artist_name}_lyrics.json`
   - Each item: `{title, album, year, lyrics}`
   - Test: run for one artist, verify the JSON file exists and has content

2. Build `ingestion/build_chunks.py`
   - Function `build_chunks(artist_name) -> list[dict]`
   - Each chunk is a rich text string combining: artist, title, album, year,
     and lyrics (truncated to 3000 chars)
   - Each chunk has metadata: `{title, album, year, artist}`
   - Test: print 2-3 chunks to verify they look correct

### Phase 2 — Vector store
3. Build `rag/vectorstore.py`
   - Use `SentenceTransformer("all-MiniLM-L6-v2")` for embeddings (runs locally)
   - Use `chromadb.PersistentClient(path="./chroma_db")`
   - Function `index_artist(artist_name)`: embeds all chunks and stores in ChromaDB
     - Collection name: `artist_name.lower().replace(" ", "_")`
     - Check if already indexed before re-indexing (avoid duplicates)
   - Function `retrieve(artist_name, query, n_results=4) -> list[str]`
     - Embeds the query, searches ChromaDB, returns top N chunks
   - Test: index one artist, then call retrieve() with a test question, 
     print the returned chunks and verify they are relevant

### Phase 3 — RAG chain
4. Build `rag/chain.py`
   - Function `ask(artist_name, question) -> str`
   - Calls `retrieve()` to get relevant chunks
   - Builds a prompt with: system context, retrieved chunks, user question
   - Calls Gemini Flash API
   - Returns the answer as a string
   - The prompt must instruct the model to only use provided context (no hallucination)
   - Test: ask 2-3 different questions, verify answers are grounded in actual lyrics

### Phase 4 — API
5. Build `api/main.py`
   - FastAPI app with CORS enabled (allow all origins for dev)
   - `POST /chat` — body: `{artist: str, question: str}`, response: `{answer: str}`
   - `POST /index` — body: `{artist: str}` — triggers indexing for a new artist
   - `GET /health` — returns `{status: "ok"}`
   - Test: run with `uvicorn api.main:app --reload`, call /health, then /chat

### Phase 5 — Deployment
6. Deploy the FastAPI backend to Render (free tier) and verify it works end-to-end.
   The Svelte integration is handled separately in the portfolio repo.
   - Push to GitHub
   - Create a Render Web Service, connect the repo, set env vars
   - Build: `pip install -r requirements.txt`
   - Start: `uvicorn api.main:app --host 0.0.0.0 --port 8000`
   - Test: curl the live URL and confirm you get a real answer

## Code style constraints
- **Always use `python-dotenv`** and `load_dotenv()` — never hardcode keys
- **Type hints on all functions** — `def ask(artist_name: str, question: str) -> str:`
- **One responsibility per file** — ingestion files only ingest, RAG files only RAG
- **Handle API errors** — Genius API can fail, wrap in try/except
- **Never re-index** if ChromaDB already has the collection — check first
- **Log clearly** — use `print()` with descriptive messages during ingestion steps
- **No notebooks** — everything in .py files, importable and testable

## Key design decisions to understand

### Why sentence-transformers locally?
No API calls, no cost, no rate limits. `all-MiniLM-L6-v2` is small (80MB) and 
fast enough for a portfolio project. Trade-off: slightly lower quality than 
OpenAI embeddings, but invisible at this scale.

### Why truncate lyrics to 3000 chars?
Embedding models have a token limit. Very long lyrics get cut off by the model 
anyway, so explicit truncation avoids silent errors. 3000 chars ≈ 600 tokens, 
well within limits.

### Why ChromaDB and not Pinecone/Weaviate?
For a local portfolio project: zero setup, zero cost, runs as a Python library. 
For production you'd swap to a hosted vector DB — the code change is minimal.

## Common issues and fixes
- **Genius page noise in lyrics** — strip everything before `[Paroles de "..."]` with regex
- **ChromaDB duplicate IDs** — use `get_or_create_collection` and check 
  existing IDs before adding
- **Gemini rate limit** — add a 1s sleep between API calls during batch operations
- **sentence-transformers first download** — the model downloads on first use (~80MB), 
  this is normal, just takes a minute
